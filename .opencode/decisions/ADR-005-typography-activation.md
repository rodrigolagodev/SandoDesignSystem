# ADR-005: Typography Activation — Two-Bridge Model

**Status:** Accepted
**Date:** 2026-06-14
**Author:** sando-architect
**Affects:** packages/tokens (build + exports), packages/components/src/styles/reset.css.ts, packages/tokens/src/recipes/{alert,card,dialog,form,tooltip}.json
**Priority:** P0

---

## Context

Audit triggered by a regression report: components render with the wrong typography (e.g. Times instead of Source Sans 3) and changing the Storybook Flavor toolbar has no visible effect on most components, even though `data-flavor` updates correctly and Flavor custom properties exist.

Investigation found that the token system declares every semantic property the design language requires:

- Flavors expose `--sando-font-family-body`, `--sando-font-family-heading`, `--sando-color-text-body`, `--sando-font-lineHeight-body`, etc.
- 16 of 21 text-rendering recipes (button, input, tag, …) declare a `fontFamily` token that references the Flavor layer.
- Custom properties cascade from Light DOM into Shadow DOM as expected.

But **no CSS rule applies any of those custom properties to a real CSS property** anywhere in the dist output:

```bash
grep -rE "^\s*font-family:" packages/tokens/dist/sando-tokens/css/
# 0 results
```

The variables exist but nobody reads them. Two symptoms result:

1. **Light DOM content** (MDX prose, app shell HTML, slot content) falls through to the browser's default font and color. The Storybook canvas has only `background-color` declared in `preview-head.html` — no `font-family`, no `color`.
2. **Component Shadow DOM** that doesn't apply its own recipe-level fontFamily inherits from Light DOM via `resetStyles`'s `font-family: inherit` — which resolves to browser defaults. Specifically:
   - 5 components have no recipe `fontFamily` AND no application: `alert`, `card`, `dialog`, `form`, `tooltip`.
   - 1 component (`option`) referenced `--sando-select-fontFamily` (cross-layer violation of TA-CR-R1).

## Decision

Introduce a **two-bridge activation model** so the existing token graph actually reaches the user's eyes. One bridge per DOM scope.

### Bridge 1 — Light DOM preflight (`@sando-ds/tokens/css/base`)

A new auto-generated file `dist/sando-tokens/css/base.css` exported as `./css/base` from `@sando-ds/tokens`:

```css
:where(:root, [data-flavor]) {
  font-family: var(--sando-font-family-body);
  font-size: var(--sando-font-size-body);
  font-weight: var(--sando-font-weight-body);
  line-height: var(--sando-font-lineHeight-body);
  color: var(--sando-color-text-body);
  background-color: var(--sando-color-background-base);
}
```

- `:where()` keeps specificity at 0 so any consumer rule trivially overrides it.
- Targeting `[data-flavor]` (not only `:root`) means a subtree flavor swap (e.g. `<section data-flavor="strawberry">`) re-applies typography automatically — no extra rules needed.
- Generated by `packages/tokens/build/utils/barrel-generator.js` so it always exists after `pnpm tokens:build`.

### Bridge 2 — Shadow DOM reset (`resetStyles`)

`packages/components/src/styles/reset.css.ts` now reads the same semantic Flavor tokens on `:host`:

```css
:host {
  font-family: var(--sando-font-family-body, inherit);
  line-height: var(--sando-font-lineHeight-body, inherit);
  color: var(--sando-color-text-body, inherit);
  font-size: inherit;
}
```

- Slot content lives in Light DOM but inherits from the host. One `:host` rule covers both Shadow DOM descendants and slotted children.
- The `inherit` fallback preserves prior behavior when no Flavor is active.
- `font-size: inherit` is intentional: applying a token value would clobber consumer overrides like `sando-button { font-size: 1.25rem }`.

### Recipe completeness (Phase 2, related)

The 5 components missing a `fontFamily` recipe token (`alert`, `card`, `dialog`, `form`, `tooltip`) now declare one. Components that own typography contracts on inner regions (e.g. `card.heading.fontFamily`, `card.body.color`) get explicit recipe tokens so each component is self-describing instead of relying on reset inheritance.

The `option` cross-layer reference is corrected to use `--sando-option-fontFamily` (already declared in its recipe).

## Options Considered

### Option A — Only fix recipes + component styles (no preflight)

Add `fontFamily` to 5 recipes, apply `font-family: var(--sando-<comp>-fontFamily)` in each component's styles.

- Fixes Shadow DOM typography for those 5 components.
- Does NOT fix slot content inheritance (slot is Light DOM child of host; without typography on the host or in Light DOM nothing applies).
- Does NOT fix MDX prose / page shell typography — every consumer must write their own `body { font-family: var(--sando-font-family-body); }` rule.
- High maintenance: each new text-rendering component needs the same edits.

### Option B — Only fix `resetStyles` (no preflight, no recipe edits)

Make `:host` read flavor tokens. Slot content inherits via host → covered.

- Fixes Shadow DOM AND slot content.
- Does NOT fix MDX/page shell text outside any component.
- Leaves the 5 recipes architecturally inconsistent — when a future engineer wants per-component typography (e.g., dialog with heading font on title), the recipe tokens don't exist yet.

### Option C — Only ship a preflight (no reset changes, no recipe edits)

Tokens package exports `base.css` for consumers to load in Light DOM.

- Fixes MDX/page shell text.
- Fixes Shadow DOM typography because `font-family: inherit` (current resetStyles) now resolves to the preflight value.
- Slot content covered.
- Does not give components a way to override typography on the host (e.g., dialog wanting heading font on title) without recipe tokens.

### Option D (chosen) — Both bridges + recipe completion

Apply A + B + C together. Each layer is independently useful and covers a gap the others can't:

- Preflight handles non-component content.
- Reset handles components even when consumer forgets to load preflight (degrades gracefully via `inherit`).
- Recipe tokens make components self-documenting and allow per-component overrides without touching reset.

## Consequences

### Positive

- Flavor switching produces visible typography changes everywhere (canvas, prose, components, slot content) for the first time.
- Every text-rendering component is self-describing at the recipe layer.
- New text-rendering components inherit the default flavor typography via `resetStyles` with zero per-component work — adding a recipe `fontFamily` is now an opt-in for overrides, not a requirement.
- Architecture documented in CA-CR-R6 and TSYS-LDA so future contributors understand the model.
- `:where()` specificity choice means consumer overrides remain trivial.

### Negative

- Consumers using `@sando-ds/tokens` directly (without `@sando-ds/components`) must remember to load `@sando-ds/tokens/css/base`. The README and TYPOGRAPHY_SYSTEM guideline document this; no automatic enforcement.
- The Shadow DOM reset now reads Flavor tokens, so a component embedded in a host without flavor tokens loaded will use `inherit` (browser default). Same end behavior as before this ADR, but the cause is now "no tokens" rather than "no rule reads them."
- `font-weight: 400` is now applied at the document root. Consumers with their own typography stack may see a regression if they assumed a different default weight; fixable via a single override.

### Migration

- No code changes required for consumers of `@sando-ds/components` — `resetStyles` change is internal.
- Consumers of `@sando-ds/tokens` should add `import '@sando-ds/tokens/css/base'` after their flavor/theme import.
- Storybook (`apps/docs`) loads it automatically via `preview.js`.

## Verification

- `pnpm --filter @sando-ds/tokens build` regenerates `base.css`.
- `pnpm --filter @sando-ds/tokens test` validates layer rules (Recipes → Flavors → Ingredients).
- `pnpm --filter @sando-ds/components test` validates component behavior unchanged.
- Storybook smoke test: switch Flavor toolbar between `sando` / `strawberry` / `kiwi` and confirm typography changes on the canvas AND inside components (especially Alert, Card, Dialog).
