# Color

The Sando Design System provides **17 chromatic palettes + 4 neutral variants + 3 utility colors** = **234 tokens**, all defined in **OKLCH color space** for perceptual uniformity. Every shade at a given step shares the same perceived lightness — `orange-500` and `blue-500` look equally bright because both use `L=0.64`.

## Color Philosophy: Warm Precision (精密な温もり)

Sando's color system follows the **Warm Precision** brand direction — the discipline of a kaiseki restaurant paired with the warmth of handcrafted ingredients.

- **Discipline**: Algorithmic lightness curves ensure every palette is perceptually consistent. No hand-tuned exceptions.
- **Warmth**: Neutral Warm (Washi 和紙) and Amber (Kohaku 琥珀) anchor the brand's warm personality.
- **Restraint**: Three chroma tiers control saturation with intent, not impulse.
- **WCAG AA Guaranteed**: All palettes are designed for accessible contrast ratios.
- **11 Shades**: From 50 (near-white) to 950 (near-black), every palette follows the same lightness curve.

**Three-layer architecture:**

1. **Ingredients** (this page) — 17 chromatic palettes + 4 neutral variants + 3 utility = 234 raw color tokens
2. **Flavors** — Map palettes to semantic roles (action, state, background, text)
3. **Recipes** — Component-specific tokens referencing Flavors

## Usage Guidelines

### Do's

- Use Ingredient tokens when building new Flavor tokens (Layer 2 references Layer 1)
- Use neutral palettes for text, backgrounds, borders, and UI structure
- Combine color with non-color indicators (icons, labels) for accessibility
- Use the universal lightness scale to guarantee consistent contrast across palettes

### Don'ts

- Don't use Ingredient tokens directly in components — use Recipes (Layer 3) instead
- Don't hardcode OKLCH values — always reference the CSS custom property
- Don't mix color systems — stay within the Sando token architecture
- Don't assume semantic meaning at this layer — "red" is not "error" until a Flavor says so

> **Note:** State colors (error, success, warning, info) are defined at the **Flavor layer**, not here. Ingredients are brand-agnostic primitives. A Flavor decides which chromatic palette maps to each state.

### Why OKLCH?

- **Perceptual uniformity**: `L=0.64` looks equally bright for every hue — unlike HSL where `L=50%` varies wildly
- **Predictable contrast**: Any `color-700` text on any `color-50` background has the same contrast ratio
- **Algorithmic generation**: New palettes can be created with just a hue angle and chroma tier
- **Gamut mapping**: OKLCH gracefully handles out-of-sRGB colors in modern displays

## Accessibility

All color combinations in the Flavors layer meet **WCAG 2.1 AA** standards:

- **Normal text** (< 18px): Minimum 4.5:1 contrast ratio
- **Large text** (>= 18px): Minimum 3:1 contrast ratio
- **UI components**: Minimum 3:1 contrast ratio
- **Focus indicators**: Minimum 3:1 contrast ratio

The uniform lightness curve guarantees that contrast ratios are consistent across all palettes. A `color-700` on `color-50` background always provides the same contrast, regardless of hue.

**Colorblind-safe by design:** Because OKLCH separates lightness from chroma and hue, all palettes maintain sufficient lightness contrast even when hue perception is impaired. Always combine color with non-color indicators (icons, text labels, patterns) for critical state communication.
